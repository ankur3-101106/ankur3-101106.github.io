Lecture 41:
Universal Hashing​

Universal Hashing (Idea)​

  • Avoid worst-case inputs by randomly choosing hash function at runtime

Definition​

A family of hash functions ( H\mathcal{H}H ) is universal if:

Pr⁡h∈H[h(x)=h(y)]≤1m,x≠y\Pr_{h \in \mathcal{H} }[h(x) = h(y)] \le \frac{1}{m}, \quad x \ne yh∈HPr​[h(x)=h(y)]≤m1​,x=y

Why Useful​

  • Prevents adversarial inputs
  • Guarantees low expected collisions
  • Works even if input is malicious

Collision Indicator​

Define:

Cxy={1h(x)=h(y)0otherwiseC_{xy} = \begin{cases} 1 & h(x) = h(y) \\ 0 & \text{otherwise}\\ \end{cases}Cxy​={10​h(x)=h(y)otherwise​

Expectation​

E[Cxy]=Pr⁡[h(x)=h(y)]=1mE[C_{xy}] = \Pr[h(x)=h(y)] = \frac{1}{m}E[Cxy​]=Pr[h(x)=h(y)]=m1​

Expected Collisions for Key (xxx)​

Let set size = (nnn)

E[collisions with x]=n−1m≈αE[\text{collisions with } x] = \frac{n-1}{m} \approx \alphaE[collisions with x]=mn−1​≈α

(where α=nm\alpha = \frac{n}{m}α=mn​ )

Key Result​

E[collisions]≤αE[\text{collisions}] \le \alphaE[collisions]≤α

✔ Small load factor → fewer collisions

Construction of Universal Family​

Example form:

  • Parameters:

    • (ai)(a_i)(ai​) chosen randomly
    • Each (ai∈0,1,…,m−1)(a_i \in { 0,1,\dots,m-1 } )(ai​∈0,1,…,m−1)

If there are (r) parameters:

∣H∣=mr|\mathcal{H}| = m^r∣H∣=mr

Bad Hash Functions​

  • Cause:

    h(x)=h(y)h(x) = h(y)h(x)=h(y)

  • Count of such “bad” functions:

≤mr−1\le m^{r-1}≤mr−1

Probability of Bad Choice​

mr−1mr=1m\frac{m^{r-1}}{m^r} = \frac{1}{m}mrmr−1​=m1​

Two-Level (Perfect Hashing)​

Idea​

  • First level:

    • Distribute keys into buckets

  • Second level:

    • Build separate hash table for each bucket

Property​

For bucket with nin_ini​ keys:

Expected collisions∝ni2\text{Expected collisions} \propto n_i^2Expected collisions∝ni2​

Guarantee​

  • Total space:

    O(n)O(n)O(n)

  • Expected collisions:

Key Insight​

  • Randomization removes worst-case
  • Universal hashing → expected O(1)O(1)O(1) operations
  • Perfect hashing → collision-free lookup

    • Collision resistant

✔ Used in all modern cryptographic systems

Lecture 43:
Secure Hash Algorithm (SHA)​

Secure Hash Algorithms​

SHA Family​

  • Developed by NIST (1993)
  • SHA-1 later replaced older versions

Variants​

  • SHA-1 → 160-bit
  • SHA-256 → 256-bit
  • SHA-384 → 384-bit
  • SHA-512 → 512-bit

Message Preprocessing (SHA-1)​

Step 1: Padding​

  • Append:

    1 followed by 0s1 \text{ followed by } 0\text{s}1 followed by 0s

  • So that length ≡ 448 mod 512

Step 2: Append Length​

  • Add original length (64-bit)

Step 3: Blocks​

  • Divide message into: 512-bit blocks512\text{-bit blocks}512-bit blocks

Initialization​

  • Initial hash values (IV): H0,H1,H2,H3,H4H_0, H_1, H_2, H_3, H_4H0​,H1​,H2​,H3​,H4​

Processing (Per Block)​

  • 80 rounds

  • Uses:

    • Logical functions
    • Constants
    • Message schedule

Compression​

For each block:

CVi+1=Compress(CVi,Mi)CV_{i+1} = \text{Compress}(CV_i, M_i)CVi+1​=Compress(CVi​,Mi​)

Final Hash​

Digest=CVL\text{Digest} = CV_LDigest=CVL​

Birthday Attack​

  • Based on probability of collisions

Probability Idea​

  • Want: P(collision)>0.5P(\text{collision}) > 0.5P(collision)>0.5

Approximation​

P≈1−e−k22nP \approx 1 - e^{-\frac{k^2}{2n}}P≈1−e−2nk2​

Solve for k​

k22n≈ln⁡2\frac{k^2}{2n} \approx \ln 22nk2​≈ln2 k≈2nln⁡2≈nk \approx \sqrt{2n \ln 2} \approx \sqrt{n}k≈2nln2​≈n​

Example (Birthday Problem)​

  • Total days:

    n=365n = 365n=365

  • Required people:

    k≈23k \approx 23k≈23

✔ >50% chance of same birthday

Implication in Hashing​

  • For n-bit hash:
Collision effort≈2n/2\text{Collision effort} \approx 2^{n/2}Collision effort≈2n/2

Example​

  • SHA-1 (160-bit):
280 operations for collision2^{80} \text{ operations for collision}280 operations for collision

Key Insight​

  • Collisions easier than brute force
  • Due to: Birthday paradox\text{Birthday paradox}Birthday paradox

Final Notes​

  • SHA-1 → broken (collision attacks exist)

  • Use:

    • SHA-256 or higher

✔ Stronger security against collisions

More on Key Exchange Protocol​

3-Party Key Exchange (Basic Idea)​

  • Parties: Alice, Bob, Charlie

  • Public:

    g,;pg,; pg,;p

  • Secrets:

    a,;b,;ca,; b,; ca,;b,;c

Exchange​

  • Alice sends:

    gag^aga

  • Bob sends:

    gbg^bgb

  • Charlie sends:

    gcg^cgc

Goal​

  • Shared key: K=gabcmod  pK = g^{abc} \mod pK=gabcmodp

Problem​

  • With normal exponentiation:

    • Requires multiple rounds
    • Not efficient

Bilinear Pairing Setup​

  • Groups:

    G1 (additive),G2 (multiplicative)G_1 \text{ (additive)}, \quad G_2 \text{ (multiplicative)}G1​ (additive),G2​ (multiplicative)

  • Pairing:

    e:G1×G1→G2e: G_1 \times G_1 \rightarrow G_2e:G1​×G1​→G2​

Properties​

1. Bilinearity​

e(aP,bP)=e(P,P)abe(aP, bP) = e(P,P)^{ab}e(aP,bP)=e(P,P)ab

2. Non-degeneracy​

e(P,P)≠1e(P,P) \ne 1e(P,P)=1

3. Computability​

  • Efficient to compute

Public Setup​

  • Generator:

    P∈G1P \in G_1P∈G1​

  • Public:

    P,;G1,;G2,;eP,; G_1,; G_2,; eP,;G1​,;G2​,;e

Private Keys​

a,;b,;ca,; b,; ca,;b,;c

Exchange (1 Round)​

  • Alice sends:

    aPaPaP

  • Bob sends:

    bPbPbP

  • Charlie sends:

    cPcPcP

Key Computation​

Alice​

K=e(bP,cP)a=e(P,P)abcK = e(bP, cP)^a = e(P,P)^{abc}K=e(bP,cP)a=e(P,P)abc

Bob​

K=e(aP,cP)b=e(P,P)abcK = e(aP, cP)^b = e(P,P)^{abc}K=e(aP,cP)b=e(P,P)abc

Charlie​

K=e(aP,bP)c=e(P,P)abcK = e(aP, bP)^c = e(P,P)^{abc}K=e(aP,bP)c=e(P,P)abc

Shared Key​

K=e(P,P)abcK = e(P,P)^{abc}K=e(P,P)abc

✔ Same for all three

Key Insight​

  • Pairing enables:

    multiplication of exponents across parties\text{multiplication of exponents across parties}multiplication of exponents across parties

  • Achieves:

    • 3-party key agreement
